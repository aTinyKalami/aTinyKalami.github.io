Github上部署Hexo框架的博客的亲测爬坑攻略

十二亚晖 2025-10-25 09:00:00
Categories： 技术 Tags： Hexo 博客 教程

Github上部署Hexo框架的博客的亲测爬坑攻略

第一部分：在Git上搭建一个标准的Hexo风格的博客

感谢这部分参考的这位作者 → 点我。

感谢另一个后面我配置SSH认证的作者也有一篇类似的指导 → 点我。

1.基础环境准备

在正式开始前，准备好：

1.一个 GitHub 账号，可以直接去官网注册：https://github.com

2.安装 Git、Node.js

Git安装方法很多，最简单的是直接官网下载软件包，然后安装（Mac 和 Windows 及其他版本都有）。

(https://git-scm.com/downloads)

Node.js也可以直接去官网下载对应系统安装包，然后一步步安装即可。

（https://nodejs.org/zh-cn/download）

下载好后直接点击软件包，默认会给我们安装好 Node.js 和 npm（Node.js 的包管理工具）。

Windows 系统可能需要自己设置环境变量，将对应安装目录添加到系统环境变量即可，浏览器搜索下对应方法即可。

3.查看下 Git、Node.js 版本，确保安装无误。Git Bash命令如下：

git --version
node -v  
npm -v

（实际这里我没有安装，前面折腾的时候装好了）

2.安装 Hexo

首先在电脑本地硬盘中新建一个项目目录，比如 HexoBlog。

然后，Git bash切换到对应文件夹：

cd HexoBlog

接下来安装 Hexo：

npm install -g hexo-cli

3.初始化一个 Hexo Blog

初始化的命令格式为hexo init <项目名称>，这里我们暂且叫做 blog。

hexo init blog

成功初始化后，会在当前文件夹下生成一个新的文件夹blog，目录结构如下：

├── _config.landscape.yml：主题配置文件，如果

├── _config.yml：站点配置文件，对站点进行配置。

├── node_modules：用来存储已安装的各类依赖包。

├── package-lock.json：包版本依赖锁文件。

├── package.json：应用数据，Hexo的版本以及相关依赖包的版本等。

├── scaffolds：博客模版文件夹，包含page.md，post.md，draft.md三种。

├── source：资源文件夹，存放静态资源如博客md文件、图片等。

└── themes：主题文件夹，Hexo通过将网站内容与主题组合来生成静态网站。

初始化项目后，我们只需在本地执行下面命令即可在本地进行预览。

首先我们需要进入到新建的项目目录下，然后执行hexo s即可启动项目，命令如下：

cd blog
hexo s

然后我们访问http://localhost:4000/即可查看网站啦。

目前是默认的主题，另外还有一篇默认的博客Hello World。

4.Hexo 基本配置介绍修改

【必改部分】 站点的基本设置，首页标题、子标题、简介、关键词（英文,隔开）、作者、语言和时区，都需要修改，这样才算自己的网站嘛。

title: 你自己的博客
subtitle: '你的人生格言'
description: '你想拿这个博客来干啥，比如我就是做长期笔记本'
keywords: 生活,学习,工作,吐槽,记录,归档
author: 你自己
language: zh-CN
timezone: 'Asia/Shanghai'

清楚缓存，然后重启 Hexo 服务，刷新页面，上面的修改就生效了。命令如下：

hexo clean 	#清除缓存
hexo s	#启动服务器。

至此博客基础风格的页面可以正常显示。

5.将 Hexo 博客部署到 GitHub

接下来，我需要将本地文件夹的文件全部同步到云端代码仓。然后这部分出了一个插曲，充分体现了无知和茫然：

我看到前面用https地址成功了，然后看到云端我的代码仓（原来装过Jekyll框架的博客，搞乱后清空重建了）里也有文件了。然后我惊奇的发现传上来的文件跟我本地的代码仓文件全部都不一样，这是为什么？

无奈的问DeepSeek，机器告诉我：这是因为你遇到了Hexo部署的一个核心概念混淆！
问题根源是：hexo deploy 上传的不是我的Hexo源码，而是生成的静态网站文件！

我本来理解Hexo和前面用的Jekyll一样，代码和静态页面是一起的。所以本地项目文件夹和云上代码仓同步后，应该两边完全一样。结果发现推上去的文件夹和本地完全不同。于是DeepSeek大法，被机器教育了一番后又get到新知识。

Hexo框架的运作方式是本地开发完后要发布成静态HTML页面，然后推到Github代码仓，就可以在博客网址显示。而核心的文档、图片即配置文件，都不会作为网页发布。

我以为的：本地Hexo项目文件夹 –部署–> GitHub仓库（应该一模一样）

实际发生的：本地Hexo项目文件夹 –生成–> public/文件夹（静态HTML） –部署–> GitHub仓库

所以在Hexo框架下，用户需要分别管理两个东西：

  1. 网站文件（自动部署）

这些通过 hexo deploy 自动管理，用户访问的就是这些文件。

  1. 源码文件（手动推送）

这些需要用户用Git手动推送到GitHub。

然后我百度了一下，针对Hexo的这种模式，有几种处理方案，最推荐的是在Github上使用两个仓库。一个私有仓库存放Hexo源码，另一个 你的用户名.github.io仓库只存放生成的网站文件。

所以我决定采用这个方案。

插曲结束，后面是基于这种两个代码仓分离的操作。

  1. 构建Github代码仓。

在 GitHub 上新建一个项目：https://github.com/new，用于托管博客静态页面，权限选公开。项目名称格式需要注意，必须是用户名.github.io。

然后再新建一个项目代码仓，权限可以选私有，用于托管博客源码。名称你自己定，比如：blog code folder。

接下来，通过Git bash进行初始化并把本地的项目代码推到云端仓库。

进入你的Hexo项目目录后，

初始化Git（如果之前初始化过，先删除.git文件夹）

rm -rf .git  # 删除所有git历史，如果存在的话
git init	#创建全新的代码仓

添加所有源码文件

git add .

提交更改

git commit -m “初始提交: Hexo博客源码”

连接到新的源码仓库

git remote add origin https://github.com/你的用户名/blog-code-folder.git

推送到新仓库

git branch -M main
git push -u origin main

此时：
输入 git push -u origin main，系统报错，说我的blog-code-folder 仓库里已经有文件了，查看网页侧发现是README.md，采用强制推送来覆盖。这里也说明，下次开代码仓做Hexo网站托管的，不要选Readme。

强制推送命令：强制用本地文件覆盖远程仓库

git push -u origin main --force

强制推送成功， blog code folder 看到所有的 线下blog 文件夹下的文档。

6.创建脚本，忽略静态文件 （可选）

为了避免把生成的静态文件也推送到源码仓库，创建 .gitignore 文件。

在git中运行：

创建.gitignore文件

cat > .gitignore << EOF

Hexo忽略文件

public/
.deploy_git/
db.json
node_modules/
*.log
.DS_Store
Thumbs.db
EOF

添加到Git

git add .gitignore
git commit -m "添加.gitignore"
git push origin main

第二部分：配置本地到云端代码仓的SSH认证

这一部分的参考文章→点我 | 感谢原作者。
参考文章原作者还有从头开始部署Hexo的文章，可以点这里。

1.设置SSH keys

1.在Git Bash输入以下指令（任意位置点击鼠标右键），检查是否已经存在了SSH keys。

ls -al ~/.ssh

2.如果不存在就没有关系，如果存在的话，直接删除.ssh文件夹里面所有文件：

3.生成密钥：

git config --global user.email "429359876@qq.com" 	# 填写你github注册并且验证的邮箱

#注意上面的 - 是两个短连线 --，原文章中的一个不对。 

git config --global user.name "github用户名" 	# 你的github 用户名

ssh-keygen 	# 会出现下面的内容，一直按Enter键就行

Generating public/private rsa key pair.
Enter file in which to save the key (/home/logan/.ssh/id_rsa):
/home/logan/.ssh/id_rsa already exists.
Overwrite (y/n)? #因为我已经生成过了所以提示我，你就一直按就行

最后在你的.ssh 目录下面生成 id_rsa(私钥) id_rsa.pub(公钥)俩个文件

第二行会提示你在哪个目录下面生成文件。

4.打开.ssh/id_rsa.pub，全选复制Key

5.配置到Github里

6.测试：

ssh -T git@github.com

遇到警告输入“yes”，
最后输出 You’ve successfully authenticated 表示添加key 成功。

7.修改yml配置文件

编辑_config.yml文件

修改文件里面的deploy部分：

deploy:
		type: git
		repo: git@github.com:jasmine-na/jasmine-na.github.io.git
		branch: main

8.完成部署

键入指令：

hexo clean	#清缓存
hexo generate #生成静态页面
hexo deploy	#部署

也可以一行搞定：

hexo clean &&hexo generate &&hexo deploy

在浏览器输入：http://你的git账号.github.io/，就可以看到你的博客了。

其他：Hexo的（Front Matter）格式标准开头

Hexo的Markdown文件标准开头（Front Matter）格式如下：

必需字段：

title: 文章标题

date: 发布时间（格式：YYYY-MM-DD HH:mm:ss）

常用可选字段

tags: 标签（数组格式）

categories: 分类

description: 文章描述

cover: 封面图片路径

注意：前后必须用 — 包围，使用YAML语法。

---
title: 我的第一篇Hexo文章
date: 2024-01-25 14:30:00
tags: [Hexo, 博客, 教程]
categories: 技术
description: 这是文章的简短描述
cover: /images/cover.jpg  # 文章封面图
---

这个时候，基本可以使用了，整体存在2中场景，常用操作如下：

1.【本地→云端 文件同步加页面发布】当我在自己的本地电脑写了新文章以后:

1.生成静态页面并通过Hexo部署网站（即发布到 .io 仓库）

hexo clean && hexo generate && hexo deploy
  1. 推送源码文件夹到 blog-code-folder 仓库，包括同步文章、配置变更等到云端:
git add .	#添加新文件
git commit -m "本地内容变更"	#提交更改
git push origin main	#推到远端代码仓

【本地 → 云端】命令合成一行代码就是：

git status && git add . && git commit -m "本地内容变更" && git push origin main
hexo clean && hexo generate && hexo deploy

2.【云端→本地 文件同步加页面发布】如果我在Github网页侧更新代码托管仓中的Source文件夹的文章内容

1.回到家将网页上写的文章同步到本地:

进入你的Hexo项目文件夹:

cd blog	#切换目录到项目文件夹

拉取远程最新更改到本地:

git pull origin main #拉取云端最新文件到本地

所以在网页上做的修改，没法通过Hexo的命令快速发布到云端？

只能每天白天改文档，晚上回到家发布，

【云端→本地】命令合起来一段代码就是：

git pull origin main	#拉取云端文件
hexo clean && hexo generate && hexo deploy	#Hexo生成静态页面并部署
