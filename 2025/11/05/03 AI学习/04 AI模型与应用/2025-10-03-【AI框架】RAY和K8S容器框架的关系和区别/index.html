调度框架 - RAY分布式调度框架解析

十二亚晖 2025-11-05 09:00:00
Categories： AI Tags： AI 技术 学习

这份文档为你严格整合了前面所有轮次的深度解析，涵盖了 Ray 的核心定义、运行机制、资源调度逻辑、与 K8S 的对比、以及在云原生环境下的计费与配合方式。这是一份系统性的“Ray 分布式调度全景指南”。

分布式计算基石：Ray 深度解析全景指南 (2026版)

一、 核心定义：应用级感知的分布式框架

Ray 是由 UC Berkeley 开发的开源分布式计算框架。它与传统调度器的最大区别在于：它不仅仅是在“找机器”，而是在**“拆代码”**。

  • 侵入式设计：Ray 通过 Python 装饰器（如 @ray.remote）直接切入 Python 代码运行时（Runtime）。
  • 应用级感知：它能“听懂” Python 代码。它知道你的 Model 对象正在加载显存，知道你的 Tensor 正在多机之间传输，从而实现极细粒度的资源分配。

二、 运行机制：Ray 为什么能“深入内部”？

Ray 能够实现秒级甚至毫秒级的任务调度，主要归功于其独特的底层组件架构。

1. 三大支撑组件

  • GCS (Global Control Service)：全知全能的大脑。存储集群所有状态（谁在运行、资源在哪、函数的代码定义）。
  • Raylet (Local Scheduler)：每台机器的星探。负责监控本地 CPU/GPU 剩余量，并快速启动 Task 或 Actor。
  • Plasma Object Store：共享内存传送门。在同一节点的不同进程间传递大数据（如 Numpy 矩阵）时，通过内存指针共享，实现零拷贝（Zero-copy），彻底消除序列化开销。

2. 编程模型：Task 与 Actor

  • Task (任务)：无状态的函数。执行完即释放资源。
  • Actor (对象)：有状态的类。实例化后常驻内存，适合加载大模型并在内存中保持，供多次推理使用。

三、 资源调度：计费、规格与自动缩容

在云平台上使用托管 Ray 服务时，资源管理是核心关注点。

1. 规格配置逻辑

配置集群规格（Head/Worker 节点）并不是设置一个简单的“上限”，而是定义计算集群的边界。Ray 通过声明式资源请求（如 num_gpus=0.5）来精准占用这些预设的硬件。

2. 计费与自动缩容 (Autoscaling)

  • 计费依据：通常按照虚拟机的开启时长计费。只要 Worker 节点处于 Running 状态，即便 CPU 闲置也会扣费。
  • 自动扩容：当代码提交的任务队列超出当前节点承载力时，Ray 自动触发云接口开机。
  • 闲置缩容 (Idle Timeout)：Ray 能感知 Python 对象是否不再被引用。一旦节点闲置超过预设时间（如 5 分钟），Ray 会主动销毁节点以停止计费。

四、 深度对比：Ray vs. Kubernetes (K8S)

虽然 K8S 也能跑 AI 任务，但它更像是一个“物业公司”，而 Ray 则是“酒店管家”。

维度 Kubernetes (K8S) Ray
调度单元 Pod (容器级)：启动慢（秒级），比较沉重。 Task/Actor (函数/对象级)：启动极快（毫秒级）。
感知深度 系统级：只看容器死活，不看内部逻辑。 应用级：感知 Python 变量、对象引用及资源细分。
数据通信 弱：Pod 间通信走网络，延迟高。 极强：内置共享内存，同节点数据交换零成本。
GPU 管理 粒度粗：通常以整块卡为单位分发。 粒度细：支持 0.1 块 GPU 等异构资源调度。
缩容依据 CPU/内存利用率（滞后且对 AI 任务不准）。 逻辑资源需求：直接根据代码任务流预判。

五、 协同作战：Ray on K8S

在企业级生产中，最佳方案通常是将 Ray 部署在 K8S 之上：

  1. K8S 负责“稳”：管理物理硬件、网络环境，当虚拟机挂掉时负责重启 Pod。
  2. Ray 负责“快”：在 K8S 提供的 Pod 资源池里，高速分发 Python 任务，处理复杂的 AI 推理流水线。

分布式计算基石：Ray 深度解析手册 (2026版)

一、 什么是 Ray？

Ray 是由 UC Berkeley 开源的分布式计算框架，旨在让开发者能够通过极其简单的 Python 代码（侵入式装饰器），将单机程序无缝扩展到成千上万个节点。

核心定位：它打破了传统大数据处理（如 Spark）和容器调度（如 Kubernetes）的边界，专门为 AI 推理、分布式训练和大规模并行任务优化。

二、 Ray 的运行机制：为什么它更“聪明”？

Ray 能够实现“应用级感知”，是因为它在 Python 运行时层面上构建了一套分布式操作系统。

1. 核心组件 (The Pillars)

  • GCS (Global Control Service)：集群的“大脑”。存储了所有任务的状态、资源的位置和代码定义。
  • Raylet (Local Scheduler)：每台机器的“监工”。负责监控本地 GPU/CPU 剩余量，并快速启动任务。
  • Plasma Object Store：基于共享内存的“传送门”。在同一台机器的不同进程间传递数据（如大矩阵）时，无需复制，直接读取内存地址。

2. 代码侵入与逻辑构建

通过在 Python 函数或类上添加 @ray.remote，Ray 会：

  1. 序列化：把你的代码打包。
  2. 构建 DAG (有向无环图)：实时分析任务间的依赖关系。
  3. 动态调度：根据代码中声明的资源需求（如 num_gpus=0.5），在毫秒级内找到最合适的机器运行。

三、 Ray vs. Kubernetes (K8S)：本质区别

虽然两者都负责“调度”，但维度完全不同。

维度 Kubernetes (K8S) Ray
调度单位 容器 (Pod)：较重，启动秒级。 函数/对象 (Task/Actor)：轻量，毫秒级。
感知深度 基础设施级：只看容器死活和基础指标。 代码级：知道 Python 变量位置、模型加载状态。
数据通信 跨 Pod 通信走网络，序列化开销大。 同节点零拷贝共享内存，极速数据交换。
GPU 管理 通常以整块卡为单位分配。 支持细粒度 GPU 切分（如 0.1 块卡）。
编程模型 运维导向：编写大量 YAML 配置文件。 开发导向：直接在 Python 代码中写逻辑。

四、 最佳实践：Ray 与 K8S 的“黄金拍档”模式

在现代 AI 生产环境中，两者通常不是二选一，而是叠加使用 (Ray on K8S)：

  1. K8S 负责“地基”：管理物理机/虚拟机资源，提供稳定的容器环境、网络和存储。
  2. Ray 负责“装修”：在 K8S 的 Pod 中运行。它接收复杂的 AI 任务，在这些 Pod 之间动态、高频地分配计算量。

收益：

  • K8S 保证稳定性：节点挂了 K8S 会重启。
  • Ray 保证灵活性：算法工程师不需要学 YAML，直接改 Python 代码就能调动千核集群。

五、 成本管理：自动缩容 (Autoscaling)

Ray 在云上的计费通常基于节点开启时长。为了省钱，必须配置 Autoscaler：

  • 按需扩容：当代码中有大量 @ray.remote 任务排队时，Ray 向云平台申请新节点。
  • 闲置缩容：当 Python 代码执行完毕，不再引用远程对象时，Ray 感知到节点空闲，触发物理机销毁（通常有 5-15 分钟的 Idle Timeout 缓冲）。

注意：必须在配置集群时设置 min_workers: 0，否则即使没任务，也会保留保底节点持续计费。

两种技术栈方案解读

具体看看这两种技术栈方案是如何运作的：

未采用Ray的传统方案

在云原生环境中构建一个复杂的分布式系统（如AutoML服务），你需要亲自处理许多底层分布式问题：

  • 应用开发：你需要用Python编写Proxy、Trainer、Worker等所有组件的业务逻辑，并自行设计它们之间的状态管理和协调机制。
  • 服务通信：需要定义gRPC通信协议和序列化方式（如Protobuf），并处理网络连接、重试、超时等复杂问题。
  • 资源调度与部署：需依赖Kubernetes进行资源调度，为每个组件编写Dockerfile和YAML配置文件。若需动态扩缩容，可能还要用Go语言开发复杂的K8s Operator。
  • 可观测性：需额外集成Prometheus和Grafana等工具来监控应用状态。

这套方案技术栈庞杂，开发者需具备多领域知识，且大量精力耗费在与核心业务无关的”分布式样板代码”上。

采用Ray后的方案

Ray的核心价值在于，它在K8s的容器编排之上，构建了一个面向应用层的统一分布式编程抽象。

  • 应用开发：你只需关注业务代码。用@ray.remote装饰器，就能将普通Python函数和类转变为分布式的Task和Actor。例如，将Worker设计为Task，Trainer设计为Actor。
  • 服务通信：Ray内置了高性能的通信层和分布式对象存储。任务间的数据传递通过ObjectRef引用完成，无需你手动处理网络通信细节。
  • 资源调度与部署：Kubernetes负责底层资源管理，而Ray负责上层应用任务调度。你可以通过ray up命令或KubeRay Operator一键部署Ray集群。Ray的Autoscaler能根据任务负载自动扩缩容计算节点。
  • 统一资源视图：Ray集群为你提供了一个统一的计算资源池。你提交任务时，只需指定CPU/GPU需求，Ray调度器会自动将其分配到合适的节点，无需关心Pod边界。

总结

简单来说，Ray并非替代K8s，而是与它分工协作：K8s继续扮演”数据中心操作系统”的角色，负责底层资源的管理和调度；而Ray则作为”分布式编程框架”，负责上层应用的计算任务调度和分布式协调。这种组合让你能更专注于业务逻辑本身，显著提升开发分布式应用的效率。
